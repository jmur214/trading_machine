1. Slippage model (Phase 1)

Goal: Simulate realistic execution prices instead of perfect fills at OHLC.

Design
	•	Add a SlippageModel abstraction, e.g. engines/execution/slippage.py:
    class BaseSlippageModel:
        def get_fill_price(self, side: str, raw_price: float, bar: dict, notional: float) -> float:
            raise NotImplementedError
    •	Start simple with config-driven models:
        •	fixed_bps: e.g. 5 bps per trade → adj_price = raw_price * (1 + sign * bps/1e4)
        •	spread_half: adj_price = ask for buys, bid for sells, estimated from high/low/close
        •	Later: impact_bps = k * (notional / ADV)

Where to plug it in
	•	In backtester/backtest_controller.py, wherever you currently compute:
    fill_price = bar["Close"]  # or similar
    replace with:
    raw_px = bar["Close"]
    notional = qty * raw_px
    fill_px = slippage_model.get_fill_price(side, raw_px, bar, notional)
    fill["fill_price"] = fill_px
    fill["price"] = fill_px
    •	This automatically flows into:
        •	Portfolio.apply_fill
        •	CockpitLogger.log_fill and your trades.csv
Config: config/backtest_config.json could include:
"slippage": {
  "model": "fixed_bps",
  "bps": 5
}
So you can easily turn it on/off.

2. Multi-edge research + AI-driven edges (Phase 2)

This is your differentiator.

a. Solidify the “edge” abstraction

Right now you already have:
	•	edge, edge_id, edge_category, and open_pos_by_edge in snapshots and trades.

Make a formal edge registry, e.g.:
class Edge:
    name: str
    category: str  # 'technical', 'fundamental', 'news', 'ai'
    def compute_signals(self, data_slice) -> dict: ...
Then your “Alpha Collector” just loops over registered edges.

b. Add new edge types
	1.	News edge
	•	External pipeline creates a daily CSV:
data/research/news_scores.csv with columns:
date, ticker, sentiment_score, source_count, headline_risk
	•	NewsEdge reads this and outputs signals like:
	•	Long if sentiment > 0.7 and volume above threshold.
	•	Short/avoid if sentiment < −0.7.
	2.	Fundamental edge
	•	Another CSV:
fundamentals.csv: date, ticker, pe, pb, roe, debt_to_equity, earnings_growth.
	•	Edges like:
	•	ValueEdge (low PE/PB)
	•	QualityEdge (high ROE, low leverage)
	•	Technical edge (what you have now) stays as is.
	3.	AI “feature” edge
	•	You compute “AI scores” offline (using ChatGPT or your own pipeline) like:
	•	“Business moat”
	•	“Management quality”
	•	“Regulatory overhang”
	•	Store in ai_features.csv: date, ticker, ai_score_1, ai_score_2, ...
	•	AI edge maps that to a numeric long/short signal.

c. Learning / weight-adjusting research engine

You already have a governor updating edge weights. Extend it to:
	•	Attribute PnL per edge using your fields:
	•	For each trade, use edge_id and realized PnL from trades.csv.
	•	For each bar, you already have open_pos_by_edge in snapshots, which gives contribution.
	•	Create a research module:
	•	Reads trades + snapshots
	•	Computes:
	•	Per-edge Sharpe
	•	Hit rate
	•	Drawdown contribution
	•	Updates edge weights in engines/engine_d_research/system_governor.py via a simple rule:
	•	e.g. exponential moving average of performance; clip between [0, 1]; renormalize.

Later, this can become more sophisticated (Bayesian updating, RL, etc.), but start with simple, transparent rules.

Health checks

	1.	Data integrity
	•	No NaNs in critical fields.
	•	No duplicate timestamps per ticker.
	2.	Portfolio integrity

	3.	PnL consistency

	•	No position > X% of portfolio unless config allows.
	•	No leverage > max_leverage.

Write a health_report.json and show a summary in the dashboard (“System Health: ✅ / ⚠️ / ❌”).

9. Market impact model (Phase 1, after slippage)

Think of this as slippage that scales with size vs liquidity.

Simple approach

In SlippageModel.get_fill_price:
	•	Look up a rough ADV (average daily volume) for the ticker (from a CSV).
	•	Compute size_pct_adv = notional / (ADV * price).
	•	Add extra bps:
    impact_bps = impact_k * size_pct_adv  # e.g. impact_k = 50
    total_bps = base_bps + impact_bps
That way big trades in illiquid names get hammered (as they should), but you don’t need to modify the whole OHLC series — just the execution price.

For paper trading, you can reuse exactly the same logic.

11. Monte Carlo simulation (Phase 1–2)

Use your actual trade results to estimate distribution of outcomes.

Implementation sketch
	•	Read trades.csv and build a series of per-trade or per-day returns.
	•	Monte Carlo module:
	•	Resample returns with replacement into synthetic paths of the same length.
	•	Do this e.g. 1000 times.
	•	For each path, compute:
	•	final return
	•	max drawdown
	•	Produce distributions (percentiles: 5th, 50th, 95th).

Expose as:
	•	monte_carlo_summary.json
	•	Dashboard chart:
	•	Histogram of final returns.
	•	Fan chart of equity paths.

This helps evaluate “possible profitability” and tail risk, not just the backtest mean.

13. Prevent “stop-hunt” dips from killing you (Phase 2)

You can’t prevent the market from dipping (sadly), but you can design more robust stop logic:
	•	Use ATR-based stops (you’re already doing ATR sizing; extend it to stops):
	•	Stop distance = k * ATR, with k higher in volatile regimes.
	•	Add time-based filters:
	•	No new stops within X minutes of the open or big macro announcements.
	•	Use close-based exits in backtests:
	•	Instead of assuming you get wicked intraday spikes, exit if close breaks your level.
	•	Implement volatility regime detection:
	•	If vol is very high, widen stops or reduce risk per trade.
	•	Consider partial take profits + trailing stop:
	•	Lock in some gains; trail the stop to reduce the frustration of being stopped at the low.

Architecturally: a StopPolicy component that takes:
StopPolicy.compute_stop(entry_price, atr, vol_regime, config) -> stop_price
And maybe StopPolicy.update_stop(current_price, atr, time_in_trade) for trailing stops.
