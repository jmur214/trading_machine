Trading Machine – Master Context

1. Vision and Mission

The Trading Machine is a modular, research‑driven quantitative trading platform written in Python.  Its goal is to evolve from a simple backtesting stack into a self‑learning, intelligent portfolio manager capable of:
	•	Ingesting and normalizing diverse financial data (OHLCV, fundamentals, sentiment, alternative data) on demand.
	•	Generating actionable trading signals via a collection of independent “edges” that encapsulate repeatable patterns in markets.
	•	Sizing positions and managing risk using configurable rules that respect capital, volatility, and portfolio constraints.
	•	Executing simulated or real orders at realistic prices with slippage, commissions and stop/target logic.
	•	Maintaining accurate portfolio accounting and recording snapshots of cash, positions, equity and PnL for every bar.
	•	Logging trades and snapshots in a durable, schema‑controlled datastore with run identifiers for reproducibility.
	•	Scoring edges based on realized performance and adapting their weights through a Governor, creating a feedback loop that learns which strategies deserve more capital.
	•	Presenting analytics, metrics and rationale to humans via dashboards and logs.

In its current form the system operates on daily bars only.  It uses Alpaca’s API (with Yahoo fallback) to fetch historical data and supports paper trading.  The long‑term vision (detailed in the System‑Architecture 2.0 design) shifts the focus from “trade execution” to portfolio management, with sleeves, target allocations, unified order models, and robust storage (DuckDB/Parquet).  The architecture is designed to remain modular so that edges, engines, and adapters can be swapped without rewriting the entire stack.

2. High‑Level Architecture

The Trading Machine is organized into several layers, each responsible for a distinct function.  The current v1 implementation follows the flow below:
DataManager → AlphaEngine (Engine A) → RiskEngine (Engine B) → ExecutionSimulator →
PortfolioEngine (Engine C) → CockpitLogger → PerformanceMetrics & EdgeFeedback →
StrategyGovernor (Engine D) → AlphaEngine
	•	DataManager fetches and caches market data, normalizes OHLCV columns, adjusts for splits/dividends, computes ATR and ensures a clean datetime index.  It uses Alpaca by default but can fall back to other sources.  Data are stored as CSV/Parquet in data/processed/.
	•	AlphaEngine collects raw scores from registered edges, normalizes and aggregates them into discrete buy/ sell/exit signals.  It enforces hygiene rules (minimum history, NaN/inf filtering, duplicate suppression), applies regime gates (trend and volatility filters) and optional ensemble shrinkage.  AlphaEngine leverages a SignalCollector to call each edge, a SignalProcessor to normalize and combine scores, and a SignalFormatter to translate aggregate scores into discrete signals.  It also enforces a flip cooldown to avoid rapid side changes.
	•	RiskEngine (Engine B) converts signals into executable orders under capital and risk constraints.  It sizes positions using ATR‑based risk budgeting or target weights and enforces per‑trade and portfolio‑level limits (max position value, max gross exposure, min trade notional, max number of positions).  It implements warmup requirements, cooldown between trades, flip logic (exit before reversing), optional enforcement of target allocations, and exposure checks.  The risk engine attaches stop‑loss and take‑profit levels to each order using configurable ATR multiples.
	•	ExecutionSimulator fills orders at the next bar’s open (with optional fallback to the close) and adds per‑side slippage and commission.  It warns about abnormal gaps and, when evaluating stops and targets intrabar, uses a conservative tie‑break rule (stop triggers before target) to prevent overly optimistic backtest bias.  It outputs a fill dictionary compatible with the PortfolioEngine.
	•	PortfolioEngine (Engine C) maintains cash, positions, realized and unrealized PnL and ensures the accounting identity equity = cash + Σ(qty × price) holds.  It applies fills from the execution simulator, updating average prices, cash, realized PnL and recording edge attribution on positions.  It supports both long and short positions and handles exits/covers, partial reductions and flips.  The engine can compute snapshots of the portfolio at any timestamp, including counts of open positions per edge, gross and net exposure, and target allocations via an optional PortfolioPolicy (not yet fully wired).  Snapshots are appended to an internal history for reference.
	•	CockpitLogger records trades and portfolio snapshots into CSV files under a unique run_id subdirectory.  It guarantees a consistent schema, auto‑adds missing columns, and buffers writes to avoid I/O bottlenecks.  Each trade row includes timestamp, ticker, side, quantity, fill_price, commission, realized pnl (for exits), edge identifiers and metadata, and the run_id.  Each snapshot row includes cash, market value, realized and unrealized PnL, equity, number of positions, open position counts by edge, and the run_id.  A background thread flushes buffers to disk periodically.
	•	PerformanceMetrics loads the snapshots and trades CSV files and computes key statistics such as returns, Sharpe ratio, max drawdown, volatility, CAGR, win rate and expectancy.  It uses epsilon floors to avoid exploding log returns when equity is near zero and can reconstruct realized PnL from trade logs using FIFO pairing if missing.
	•	EdgeFeedback (analytics/edge_feedback.py) reads the latest trade and snapshot logs, computes per‑edge PnL metrics (rolling Sharpe, max drawdown, trade count and correlation penalties) and updates the StrategyGovernor.  It applies recency decay so that recent trades have more influence and merges optional evaluator recommendations from research modules.  Updated weights are saved to data/governor/edge_weights.json, and metrics are stored in edge_metrics.json.
	•	StrategyGovernor (Engine D) maintains a rolling window of performance metrics for each edge and produces soft weights (0–1) using defense‑first policies.  Edges with low Sharpe or high drawdown are zeroed out; otherwise the Sharpe is mapped to a weight between a floor and 1.0, optionally penalized for negative correlation with total equity.  Weights are smoothed via exponential moving average and normalized to ensure they sum to ≤1.  The governor can merge evaluator recommendations from research, persist weight history and metrics, and provides a public API for retrieving and saving weights.
	•	BacktestController orchestrates the entire pipeline.  For each bar in the selected time range, it builds a history slice for each ticker and calls AlphaEngine.generate_signals().  It passes each signal to the risk engine to produce orders, executes exits before entries, fills orders via the execution simulator, applies resulting fills to the portfolio, logs the fills and snapshots, checks for stop/target triggers within the same bar and records a snapshot at the end of each bar.  It ensures that snapshots always exist (including an initial snapshot) and that run_id is propagated.  After finishing, it returns the full portfolio history.
	•	Data Storage resides in data/:
	•	data/processed/ contains cached market data files (CSV/Parquet) keyed by ticker and timeframe.
	•	data/trade_logs/ holds per‑run subdirectories of trades.csv and portfolio_snapshots.csv plus backups.
	•	data/governor/ stores edge_weights.json and edge_metrics.json capturing governor state.
	•	data/research/ contains results of parameter sweeps and evaluator outputs.

3. Data Flow and Contracts

3.1 End‑to‑End Bar Loop

For each timestamp t in the backtest:
	1.	Signal Generation (Engine A):  The controller slices each ticker’s history up to t and calls AlphaEngine, which:
	•	Uses SignalCollector to call each active edge’s compute_signals/generate_signals/generate method, converting lists or dicts into a unified mapping raw_scores[ticker][edge] = score.
	•	Passes raw scores to SignalProcessor, which normalizes each edge’s score to [−1, 1], applies regime filters (trend & vol), clamps outliers, applies optional ensemble shrinkage and computes a weighted aggregate per ticker.  The edges_detail list records raw and normalized values and weights.
	•	Passes aggregate scores to SignalFormatter, which returns (side, strength) tuples: long if score ≥ enter_threshold, short if ≤ −enter_threshold, None if |score| < exit_threshold; the zone between exit_threshold and enter_threshold yields no new signal (position persists).  Each signal includes metadata about contributing edges and regimes.
	2.	Order Preparation (Engine B):  For each signal the controller queries the portfolio for current position and passes signal, equity and the ticker’s price history to the risk engine.  The risk engine checks warmup bars, cooldown, neutral logic, flip logic, maximum positions and gross exposure.  It computes ATR (or uses fallback standard deviation) and sizes quantity based on risk per trade and ATR stop distance, or aligns to target weights if provided.  It enforces min notional and min quantity, clamps sizes if max position value is exceeded, attaches stop and take‑profit distances, and returns an order dictionary (ticker, side, qty, stop, take_profit, plus edge attribution).  If constraints fail, the method returns None and records a skip reason.
	3.	Order Execution (ExecutionSimulator):  Orders are sorted so exits happen before entries.  For each order, the simulator picks the next bar (t+1) and fills at the open (fallback to close if invalid).  It applies per‑side slippage and commissions and propagates edge identifiers and meta.  It returns a fill dictionary with fill_price, commission, trigger (entry/exit) and edge info.  After all fills, the simulator checks stops and targets intrabar within bar t+1 for each open position and triggers exits if levels are breached, using conservative tie‑breaks.  Fills from stop/target triggers include the trigger type.
	4.	Portfolio Update (Engine C):  Each fill is applied to the portfolio.  For entries, cash decreases (or increases for shorts), average price updates and edge metadata are attached.  For exits, the engine calculates realized PnL, updates cash (buy to cover or sell to close), resets or adjusts position quantity and average price, and updates realized PnL.  Stop and take‑profit levels are saved on the position.  The portfolio ensures equity = cash + ∑ qty × price holds.  After processing all fills, the engine takes a snapshot at bar t+1 using closing prices for each ticker.  The snapshot includes realized/unrealized PnL, open positions count and open_pos_by_edge counts for quick attribution.  Snapshots are appended to portfolio.history.
	5.	Logging (CockpitLogger):  Fills and snapshots are appended to internal buffers and eventually flushed to CSV.  Each row records the run_id to enable multi‑run isolation.  The logger computes realized PnL for exits on the fly if missing and provides methods to force flush.
	6.	Analytics & Feedback:  After a run (or periodically), PerformanceMetrics computes trading statistics from the snapshots and trades.  update_edge_weights_from_latest_trades() loads the latest logs, computes per‑edge metrics and calls StrategyGovernor.update_from_trades().  The governor updates weights, applies recency decay, merges external evaluator recommendations, normalizes and persists the new weights.  The updated weights feed into the next backtest or live run via AlphaEngine’s config.

3.2 Key Contracts and Invariants
	•	Data normalization:  DataManager guarantees all price data frames have Open, High, Low, Close columns (creating them if necessary), a tz‑naive datetime index sorted ascending, numeric types and computed ATR.  Median price scaling is applied when median > 1000 to avoid mis‑sized risk calculations.  Missing values and splits are handled before use.
	•	Equity accounting:  PortfolioEngine must satisfy equity = cash + Σ(qty × price) for every snapshot.  Realized PnL only changes on closes (exits or covers).  Unrealized PnL is computed as (current price − avg_price) × qty.  Cash and realized PnL are floats and should never be NaN.
	•	Order constraints:  RiskEngine enforces qty ≥ 1, order_notional ≥ min_notional, abs(qty*price) ≤ max_pos_value_pct × equity, and gross_exposure ≤ max_gross_exposure.  It also enforces max_positions (no new positions if limit reached), min_bars_warmup and optional cooldown_bars.  If allow_shorts = False, no short orders are emitted.
	•	Signal gating:  AlphaEngine uses enter_threshold and exit_threshold to translate aggregate scores into discrete signals.  The grey zone means no change; flip cooldown ensures a minimum number of bars before reversing a position.
	•	Run separation:  CockpitLogger writes each run to its own subdirectory named by run_id.  A top‑level convenience trade and snapshot file may be created by promotion logic, but the core workflow is per‑run to avoid cross‑contamination.  Edges, governor state and metrics are also versioned via run_id.
	•	Edge expectations:  Each edge module must implement one of compute_signals(data_map, now) -> dict[ticker, float] returning raw scores, or generate_signals(data_map, now) -> list[dict] returning dictionaries with keys (ticker, side, confidence or score) and optional edge_id/edge_category/meta.  SignalCollector coerces lists into ticker→score dictionaries.  Edges that return invalid formats (e.g. lists of dicts without keys) will be silently filtered, as noted in the audit report.
	•	Governor smoothing:  StrategyGovernor maps rolling Sharpe ratios to soft weights, kills edges when Sharpe < disable_sr_threshold or drawdown < disable_mdd_threshold, applies correlation penalties when enabled, warms up for a minimum number of days, and exponentially smooths weight updates.  Weights are always clamped to [0,1] and normalized to sum to ≤1.  Recency decay (λ=180 days) scales weights based on days since last trade.

4. Subsystems and Modules

4.1 Engines
	•	Engine A (AlphaEngine) – orchestrates edge collection, processing and aggregation.  It exposes generate_signals(data_map, now) which returns a list of signal dictionaries for the risk engine.  It uses runtime configuration loaded from config/alpha_settings.<env>.json, plus dynamic edge weights from the governor.  Submodules include:
	•	edge_registry.py – file‑backed registry mapping edge_id→module and storing meta (category, version, status).  It loads YAML from data/governor/edges.yml and is used by run_backtest.py to dynamically load edges.
	•	signal_collector.py – introspects edge objects or modules, calling compute_signals, generate_signals, generate or instantiating classes.  It converts lists/dicts into a unified raw score dictionary.
	•	signal_processor.py – normalizes raw scores, applies regime filters (trend/vol based on moving averages and volatility Z‑scores), hygiene filters (min history, duplicate suppression, clamp) and ensemble shrinkage.  It outputs per‑ticker aggregate scores and details.
	•	signal_formatter.py – translates aggregate scores into discrete sides and strengths using configurable enter/exit thresholds.
	•	Engine B (RiskEngine) – converts signals into orders.  Configurable via config/risk_settings.<env>.json, it defines risk_per_trade_pct, ATR stop/target multiples, clamps, max positions, max gross exposure, short allowances, min notional, rounding, and optional portfolio alignment.  The prepare_order() method returns a dict with ticker, side, qty, stop, take_profit and meta or None with a skip reason.  Skip reasons include warmup_insufficient_bars, cooldown_active, max_positions_reached, invalid_price, no_incremental_size, below_min_notional and gross_exposure_limit.  The risk engine also records the last action bar for cooldown.
	•	Engine C (PortfolioEngine) – maintains a Position dataclass per ticker (qty, avg_price, stop, take_profit, edge metadata) and cash/realized PnL.  apply_fill() handles both opening and closing trades, computing realized PnL on exits or flips and updating avg_price on adds.  snapshot() computes market value, unrealized PnL and equity at a timestamp and stores open_pos_by_edge counts.  The engine can compute total equity on demand and translate target weights into notionals via a policy (not yet fully utilized).
	•	Engine D (StrategyGovernor) – computes per‑edge metrics and adaptive weights.  Using GovernorConfig from config/governor_settings.json, it sets rolling window length, kill thresholds, maximum weight, EMA halflife, warmup days and Sharpe→weight mapping bounds.  update_from_trades() builds daily PnL series for each edge, computes pseudo returns (pnl divided by MAD), calculates Sharpe, rolling max drawdown, optional correlation penalties, and maps Sharpe to a weight.  It merges with previous weights using an EMA and normalizes.  The governor can also import evaluator recommendations from research modules.

4.2 Backtesting & Live Scripts
	•	scripts/run_backtest.py is the CLI entry point for historical simulations.  It loads configuration files (backtest_settings.json, risk_settings.<env>.json, edge_config.json, alpha_settings.<env>.json), fetches data via DataManager, dynamically loads active edges from EdgeRegistry and config, instantiates the StrategyGovernor, AlphaEngine, RiskEngine and CockpitLogger, then constructs BacktestController and executes the run.  Flags allow clearing previous logs (--fresh), enabling alpha debug (--alpha-debug), skipping governor updates (--no-governor), choosing environment (--env dev/prod), and separating sandbox/prod modes for data paths.  After the run, it can invoke PerformanceMetrics and EdgeFeedback to update weights.
	•	scripts/run_live.py (planned) will replace the ExecutionSimulator with a broker adapter (e.g. Alpaca) and reuse the same OMS to route orders to live accounts.  A unified order model defined in portfolio/order_types.py and an OMS adapter in portfolio/oms.py will support both simulation and live execution.

4.3 Analytics & Dashboard
	•	analytics/edge_feedback.py – detailed above, provides the adaptive feedback loop.
	•	cockpit/dashboard.py (not fully detailed here) – a Dash/Plotly app that displays equity curves, drawdowns, PnL by edge, KPIs, benchmark comparisons, heat maps, and optionally news summaries.  It supports multiple modes (Backtest, Paper, Live) and uses the CockpitLogger files or Alpaca API for data.  Future versions will embed TradingView charts for interactive visualization and overlay of signals and governor weights.
	•	cockpit/metrics.py – computes advanced performance metrics (Sharpe, Sortino, max drawdown, volatility, CAGR, win rate, expectancy) from snapshots and trades.  It includes FIFO pairing to compute realized PnL when not provided by the logger.

4.4 Data Management
	•	engines/data_manager/data_manager.py – handles data ingestion.  It loads .env for API keys, ensures a cache directory, and fetches historical bars from Alpaca using StockHistoricalDataClient.  It normalizes DataFrames: sets a tz‑naive datetime index, renames columns, fills missing OHLCV values, coerces to numeric, scales prices when median > 1000, computes ATR, and performs sanity cleanup.  Data is written to CSV and Parquet.  On subsequent runs the DataManager checks the cache and only refetches missing ranges.  When intraday data is later supported, DataManager will infer timeframe from file names.
	•	config/ – contains JSON files specifying backtest date ranges, tickers, initial capital, slippage bps, risk parameters per environment, alpha thresholds and edge weights, as well as governor settings.  A YAML file in data/governor/edges.yml stores registered edges.

4.5 Utilities and Helpers
	•	utils/config_loader.py – safe JSON loader/writer with friendly messages and default returns.
	•	debug_config.py – defines debug levels for various modules (e.g. ALPHA, RISK, BACKTEST) and utility functions is_debug_enabled and is_info_enabled.
	•	tests/ – PyTest suites ensure edge outputs conform to expected schema, signal collection works across formats, alpha pipeline produces consistent signals, portfolio accounting balances cash and equity, backtest controller orchestrates correctly, and governor feedback updates weights.  Continuous validation scripts run these tests periodically.

5. Edge System and Research

Edges represent independent hypotheses about market behaviour.  Each edge is implemented as a Python module in engines/engine_a_alpha/edges/ and must define either:
	1.	compute_signals(data_map: Dict[str,pd.DataFrame], now: Timestamp) -> Dict[str, float] returning a raw score per ticker, or
	2.	generate_signals(data_map, now) -> List[Dict] returning signal dictionaries with ticker, side and confidence/score along with optional edge_id, edge_group, edge_category and meta.  The collector will convert these into raw scores for aggregation and also propagate meta into final signals.

Example edges include:
	•	ATRBreakoutEdge – momentum: compute 14‑period ATR and a 20‑period price breakout; normalized breakout is passed through tanh and used as score; signals set side based on sign and confidence as absolute score; meta includes an explanation string.
	•	rsi_mean_reversion.py (rewritten as of Nov 9 2025) – uses RSI(14) and MA(200) to detect oversold/overbought conditions; generates long when RSI < lower band and price > MA, short when RSI > upper band and price < MA, exit otherwise; meta includes RSI, MA and price for dashboards; includes confidence model based on distance from 50.
	•	xsec_meanrev.py / xsec_momentum.py – cross‑sectional strategies ranking stocks by normalized returns and betting on mean reversion or momentum.  Audit notes revealed that xsec_momentum incorrectly returned a list of dicts rather than a dict and used a DataFrame input instead of a data_map; this prevented signals from propagating.  Edges must adhere to the expected API.
	•	momentum_edge.py, bollinger_reversion.py, and others implement breakouts, Bollinger band reversion, pairs trading, etc.  Each sets EDGE_ID, EDGE_GROUP, EDGE_CATEGORY constants to aid attribution.

The research module (research/edge_harness.py) runs parameter sweeps and walk‑forward tests for edges, writing results to data/research/.  An edge_db_viewer script inspects this database.  Future research will include multi‑edge fusion, feature generation from news and fundamentals, and AI‑assisted scoring.  The governor can later merge evaluator recommendations produced by these tools.

6. Future Directions and v2 Design

The System‑Architecture 2.0 design document outlines the evolution toward a production‑ready portfolio‑first platform.  Key changes include:
	1.	Unified Order Management System (OMS) – unify order, fill and position models across backtest, paper and live modes.  A portfolio/order_types.py defines canonical Order, Fill and Position dataclasses (with sides BUY/SELL/SELL_SHORT/BUY_TO_COVER, order types MARKET/LIMIT/STOP/STOP_LIMIT, time‑in‑force DAY/GTC/IOC/FOK).  A portfolio/oms.py will route orders to either a simulation adapter (wraps ExecutionSimulator) or a live broker adapter (wraps Alpaca).  A portfolio/sleeves.py will manage allocations across sleeves (core holdings, tactical edges, experimental) and enforce drift control and sleeve budgets.
	2.	Portfolio Engine v2 – becomes the single source of truth for positions, cash and equity across modes.  It will maintain sleeves, track target weights by edge and sleeve, compute drift, and generate rebalance orders.  It will provide risk hooks (beta/exposure/turnover/day‑loss) and integrate with the governor to adjust capital allocation.
	3.	Robust Data Layer – replace brittle CSV logging with DuckDB or Parquet tables keyed by run_id and validated via schemas.  Trade and snapshot logs will be stored in relational tables for easier querying and analytics.  CSV fallback will remain for human inspection until cut‑over.
	4.	Adaptive Allocation – the governor’s weights will control actual capital allocation, not just log suggestions.  A softmax or custom mapping of Sharpe to capital share will be applied, with diversification penalties based on correlation.  Sleeve budgets will enforce that, e.g., core holdings must remain ≥40% of capital, tactical edges ≤50%, experimental ≤10%.
	5.	Expanded Edge Research – the v2 design encourages new edge types: news sentiment edges, fundamental/value edges, AI feature edges, and cross‑sectional factor edges.  A formal edge registry and harness will enable parameter exploration, cross‑validation and auto‑discovery of new edges while retiring underperformers.  Research outputs will feed into the governor via evaluator recommendations.
	6.	Market Realism – incorporate slippage models (fixed bps, half‑spread, impact‑based), market impact (square‑root law), borrow costs for shorts, regime filters (conservative vs degen risk modes) and Monte Carlo simulation for stress testing.  Phase 1 of the roadmap includes implementing the slippage model interface.
	7.	Intraday Support – extend DataManager and backtester to handle multiple timeframes (5m/15m/hourly), allow edges to declare their timeframe, and modify the loop to iterate at intraday resolution.  The architecture already anticipates this by inferring timeframe from file names and not forcing daily bars.
	8.	Monitoring and UX – build a more robust Cockpit dashboard (v2) with tabs for recommended trades, health/system checks, and context documents.  Integrate TradingView Charting Library for interactive charts and support streaming data for live mode.  Add alerting via Discord/Telegram.
	9.	Governance & Explainability – persist rationale for weight changes, exposures by sleeve and edge, and integrate explainability into dashboards.  Provide transparent logs showing why edges were reweighted and what triggered kills.
	10.	Testing & Hygiene – expand test coverage, enforce schema validation for config and data, implement CI pipeline, and maintain de‑debugging/hardening tasks.  Add tax‑aware logic (analysis mode), volatility clustering handling, and best‑practice alignment to actual PnL.

7. Current Limitations
	•	Daily‑only data – the v1 system operates on daily bars; there is no support for intraday data or multiple timeframes yet.  However, the DataManager and architecture are prepared to add them later.
	•	API fragility – some edge modules return lists of dicts or expect a DataFrame rather than a data_map; SignalCollector must coerce these and may silently drop invalid entries.  A formal edge base class with strict interface would improve reliability.
	•	CSV storage – trade and snapshot logs are stored in CSV files which can become corrupt if schema drift occurs or if multiple runs are merged.  The per‑run subdirectory scheme mitigates some issues, but the system still falls back to flat CSVs in some cases (recently fixed).  Moving to DuckDB/Parquet will improve robustness.
	•	Risk management – the risk engine uses ATR stops and simple per‑trade risk budgeting.  It does not yet handle dynamic volatility regimes, beta exposure, sector caps, leverage, or cross‑asset correlations.  Multi‑asset portfolios (stocks, ETFs, crypto, futures) are not supported out of the box.
	•	Governance – the governor uses pseudo returns (pnl divided by MAD) instead of true returns due to lack of per‑edge capital tracking.  It does not enforce diversification or sleeve budgets yet.  Correlation penalties are off by default.
	•	Order management – there is no unified OMS in v1; orders are simple dicts passed through the simulator.  There is no handling of limit orders, stop‑limit orders, trailing stops or partial fills.
	•	Machine learning – edges are rule‑based; there is no integrated ML pipeline yet.  Research modules mention Random Forests, XGBoost, LSTMs, reinforcement learning and meta‑learning, but these are future work.
	•	Limited universe – backtests typically run on a handful of tickers.  Expanding to the full S&P 500 or international stocks will require optimizing DataManager, risk engine and governor for scalability and ensuring that per‑bar loops remain tractable.

8. Summary

This master context document assembles knowledge from the repository, system design documents, audits and chat transcripts to provide a comprehensive understanding of the Trading Machine.  It traces the flow of data from ingestion through signal generation, risk sizing, execution, portfolio accounting, logging, analytics and adaptive governance.  It records the contracts and invariants each component must respect and highlights both implemented functionality and planned improvements.  Future AI assistants and developers should keep this document open when working on the codebase to ensure architectural coherence and avoid reinventing existing mechanisms.
