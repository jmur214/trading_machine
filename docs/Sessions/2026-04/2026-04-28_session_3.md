## What was learned

- **Bare-except is the dominant Engine D failure mode**: 6 bare `except Exception` blocks in
  `validate_candidate` alone, plus more in tree_scanner / wfo / feature_engineering. Every one
  masked at least one programmer error (TypeError, AttributeError, missing-method, missing-key,
  deprecated-API, schema-drift). The bugs were silently disabling Gate 3, Gate 5, regime context,
  hunter signals, and macro_credit_spread thresholds simultaneously. After this session,
  exception handlers in the validation path log `type(e).__name__` and re-raise programmer
  errors so future drift fails loudly.

- **Hunter candidates have a deeper feature-availability bug than `compute_signals` alone**:
  Adding `RuleBasedEdge.compute_signals()` (commit dda474c) removed the missing-method error,
  but hunters still produce Sharpe=0.00. Root cause: `check_signal()` reads features like
  `RSI_14` and `Vol_ZScore` from the row, but `validate_candidate`'s data_map has only OHLCV.
  FeatureEngineer columns are computed only during the hunt phase and never propagated to
  validation. This is a separate HIGH finding, not yet fixed.

- **Discovery validation runs the full data parquet extent, not the configured backtest
  window**: 6 years of data per candidate per gate. Logged as MEDIUM finding for future
  optimization once the loop is fully verified.

- **Engine D bugs cluster in interface-drift hot spots**: Every bug found this session
  (regime_detector typo, WalkForwardOptimizer signature mismatch, datetime-index missing,
  RuleBasedEdge method name, regime_meta key shape, pandas-2 deprecation, dead module imports)
  was at a boundary where Engine D consumes another engine or external library. Charter
  Invariant 5 ("D's research is fully reproducible") was being violated by silent gate-skip
  failures.
