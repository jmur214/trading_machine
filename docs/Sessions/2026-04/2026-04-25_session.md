# Session Summary: 2026-04-25

## What was worked on

- **Universe expansion** (39 → 109 tickers) and first lifecycle stress test on a wider universe. Smoke run showed Sharpe collapse from 0.979 → 0.332 on the expanded universe; lifecycle correctly paused both `atr_breakout_v1` and `momentum_edge_v1` based on evidence.
- **Discovered + fixed a critical autonomy bug**: `EdgeRegistry.ensure()` was silently overriding lifecycle pause/retire decisions on every module import, because auto-register-on-import code in `momentum_edge.py` and `momentum_factor_edge.py` calls `reg.ensure(EdgeSpec(..., status="active"))`. `atr_breakout_v1` escaped the bug (no auto-register block); every other lifecycle-paused edge was silently reverted to `active` on the next backtest. Fixed by write-protecting `status` in `ensure()` per the `edges.yml` Write Contract documented in `PROJECT_CONTEXT.md`.
- **First true post-pause measurement** after the registry fix: Sharpe 0.332 → 0.403 on the 109-ticker universe with both major edges genuinely paused. Modest +0.07 Sharpe lift, substantial +4.24pp MDD improvement (-13.27% → -9.03%). Lifecycle decision was directionally correct.
- **Permanent test coverage** for the autonomy machinery: 12 tests in `tests/test_edge_registry.py` (including explicit regression test for the stomp bug), 9 tests in `tests/test_lifecycle_manager.py` (covering pause / retirement / revival protection / cycle caps / audit-trail evidence).
- **First FRED-consuming edge shipped**: `engines/engine_a_alpha/edges/macro_yield_curve_edge.py`. Reads the 10Y-2Y Treasury spread from the parallel agent's macro_data cache; emits uniform regime tilt (+0.3 normal / 0 neutral / -0.3 inverted). Modest by design — regime modulator, not per-ticker alpha. Graceful when cache empty (abstains entirely). Registered active at weight 0.5; activates the moment FRED cache is populated.

## What was decided

- **Don't manually re-tune edge weights when the universe expanded and Sharpe dropped.** The user's correction: the lifecycle should rebalance autonomously. Ran the lifecycle-active backtest. The lifecycle correctly paused the 2 edges whose loss-fraction exceeded the gate. Decision validated empirically.
- **Treat bitwise-identical canon md5 when expecting change as a P0 anomaly.** Was the diagnostic that surfaced the registry stomp bug — the post-lifecycle 1-run produced canon `2f078a65...` matching the pre-lifecycle smoke test exactly, which should have been impossible. Pursued the anomaly instead of accepting it. Now codified as a methodology rule in `lessons_learned.md`.
- **Documented contracts need executable enforcement.** The `edges.yml` Write Contract ("F writes status; A doesn't touch status") had been documented from project genesis but never enforced in code. The bug fix is one line; the durable answer is the test file (`test_edge_registry.py`) that asserts the contract.
- **Don't revert universe expansion to recover the 0.979 number.** The 39-ticker baseline was a curated-mega-cap-tech artifact. Going back to it doesn't make the system better. The 109-ticker Sharpe 0.4 is the honest baseline going forward.
- **Modest starting weight (0.5) for the macro yield-curve edge.** Other "regular" edges are at 0.5-1.0 in `alpha_settings.prod.json`; the regime-tilt nature of this edge (uniform across universe, max magnitude ±0.3) means it should be a modulator not a dominator. Conservative default; lifecycle/governance can adjust based on evidence.

## What was learned

- **The 39-ticker Sharpe 0.979 was misleading.** Same edge stack, similar-quality wider universe (109 large-caps): Sharpe 0.4. The original number was concentration on a particularly favorable subset of the market — same failure mode that broke `momentum_factor_v1` walk-forward yesterday (top-quintile-of-39 = concentration risk masquerading as factor exposure). **Rule: every Sharpe claim must specify universe + period; comparing Sharpe across different universes without that context is meaningless.**
- **Audit trails record decisions, not effects.** `lifecycle_history.csv` showed pauses firing cleanly and gave a false sense of working autonomy. The actual effect (status persisting in `edges.yml`, trade behavior changing on the next run) needed to be cross-checked separately. The bug had been in the system from when momentum_edge.py first added its auto-register block.
- **Repeated identical lifecycle events for the same edge are a smell.** If `lifecycle_history.csv` shows `momentum_edge_v1: active → paused` on multiple consecutive runs, that's impossible under correct behavior — the second run should see the edge already paused. Future low-priority refinement: a sanity check at lifecycle startup that flags `<id>: <prev> → <new>` events where `prev` doesn't match the registry's actual current value.

All four rules are written into `docs/Progress_Summaries/lessons_learned.md` with the registry-stomp-bug entry.
