Behavior:
- Input frame **without** any of these columns → no `XS_<Fund>_*_Pctile` columns added (no spurious empties).
- Input frame **with** some subset → exactly those columns get rank companions.
- Per-date `rank(pct=True)` ranking — straight Pearson percentile, ties handled by pandas default (`average`).
- PIT discipline is the **caller's** responsibility. Convention follows `_fundamentals_helpers.latest_value` / `ttm_sum` (publish-date filter ≤ as-of). The cross-sectional ranker treats whatever values are present at row-i as already-PIT-aligned; it does not back-shift or look-ahead-correct.

## What this enables — forward-looking note

R1's pushback on the Bayesian-opt-replaces-GA recommendation was *"a different optimizer over the same narrow space won't help."* The vocabulary here is the search space.

Previously-impossible candidate edges Discovery's TreeScanner / GA can now reach:

1. **Cross-asset confirmation edges.** `Foundry_hyg_lqd_spread` + `Foundry_dxy_change_20d` + `Foundry_vvix_or_proxy` give Discovery the credit / dollar / vol-of-vol primitives that the existing technical block can't see at all. Edges of the shape *"buy high-momentum names ONLY when HYG-LQD spread is tight"* are now expressible.
2. **Calendar-conditional edges.** `Foundry_days_to_quarter_end` + `Foundry_month_of_year_dummy` enable end-of-quarter rebalancing-flow / window-dressing edges.
3. **Cross-sectional V/Q/A edges discovered automatically.** `XS_ROE_Pctile`, `XS_Gross_Profitability_Pctile`, etc. give Discovery the same rank primitives the hand-rolled V/Q/A edges already use. Discovery can find composite rules like *"top quintile gross profitability AND bottom quintile sloan accruals AND positive 12-1 momentum"* — i.e., quality-momentum hybrids, without anyone hand-writing them.
4. **Cross-asset-vol-regime-conditional momentum.** `Foundry_vol_regime_5_60` + `Foundry_realized_vol_60d` + `Foundry_mom_12_1` together expose the standard "momentum crashes when vol regime flips" pattern that the regime-conditional activation work has been struggling to express in the existing vocabulary.
5. **Pair-relative primitives.** `Foundry_pair_zscore_60d` opens stat-arb edge templates Discovery had no way to even propose under the technical-only vocabulary.

This is **vocabulary expansion**, not promotion. Whether Discovery's GA / TreeScanner finds anything useful in the wider space is the next discovery cycle's job. The acceptance test is *"Discovery can now reach edges it couldn't before"* — which the column-count delta confirms.

## Open questions raised by the spec

The spec asked these to be surfaced (not blocked-on); each is documented here for director triage.

### Q1 — Should Engine D respect the `horizon` field on Foundry features?

Some Foundry features expose a `horizon` (e.g. `cot_commercial_net_long` has `horizon=5`). The semantic is "this feature is only informative `horizon` bars forward — shift forward by that many bars to align with a forward-looking target."

**Current implementation:** I do **not** shift. The feature's value at bar `t` lands in row `t` of the column. This is the right call IF the consumer (TreeScanner) frames its target as `target = forward_return(t, t+H)` — the standard ML setup, where row `t` already aligns "feature-at-t with future-target". Shifting forward in feature_engineering would double-shift.

**Recommendation:** keep no-shift here. Document the convention so TreeScanner's target framing is auditable. If later the system grows multi-horizon targets, the per-feature `horizon` lets TreeScanner select features that match its target horizon.

### Q2 — Tier-B features: ingest now or wait for production validation?

Spec direction: ingest both A and B for vocabulary breadth; tier filtering happens at promotion time, not ingestion. **Followed as-spec'd.** All 24 currently-registered features are tier-B; ingesting only tier-A would have given Discovery zero new features.

The `adversarial` tier is intentionally skipped — those are auto-generated permuted twins for the leakage detector, not vocabulary.

### Q3 — Cross-sectional fundamentals percentile and PIT alignment

Spec direction: "use the as-of date convention already established in `engines/engine_a_alpha/edges/_fundamentals_helpers.py` rather than reinvent."

`_fundamentals_helpers` enforces PIT via `panel[panel["publish_date"] <= asof_ts]` *inside* the panel access primitives (`latest_value`, `ttm_sum`). The cross-sectional ranker assumes the caller has already done this. If a caller passes a non-PIT panel, the ranker happily produces look-ahead-contaminated ranks.

**Recommendation:** keep the contract "caller is responsible for PIT". Discovery's TreeScanner consumer should always join from `simfin_adapter.load_panel()` with the publish-date filter applied first. If we later see a leak, the right fix is in the joiner upstream of `compute_cross_sectional_features`, not in the ranker.
