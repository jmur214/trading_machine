3. Choosing good edges

Edges for retail systems usually fall into a few categories:

A. Behavioral / Pattern-Based
	•	Market overreactions: e.g., post-earnings moves that overshoot then revert.
	•	Reversal patterns: mean reversion in highly volatile intraday ranges.
	•	Crowd psychology: momentum fueled by retail flow.

B. Statistical / Quantitative
	•	Pairs trading / relative value: when correlated assets temporarily diverge.
	•	Seasonal / time-of-day effects: predictable small biases in asset returns.
	•	Volatility regimes: some strategies perform better under certain market conditions.

C. Event-Driven / News
	•	Earnings surprises, FDA approvals, analyst upgrades, options flow.
	•	Works well if the system reacts mechanically and quickly, but you don’t need sub-millisecond speed—minutes can be enough.

D. ML-Based
	•	Supervised ML classifiers predicting short-term up/down moves from price/volume patterns.
	•	Reinforcement learning for portfolio allocation and position sizing.
	•	Caveat: ML requires careful validation; overfitting is easy if you don’t account for regime shifts.

Engine A — Multi-Edge Combination & Scoring Logic

⸻

1. Core Design Philosophy

Every edge acts like an “analyst.”
Each edge produces an opinion about direction (+1 = long, −1 = short, 0 = neutral) and a confidence weight (0–1).
Engine A’s job is to:
	1.	Aggregate those opinions intelligently.
	2.	Filter noise (low-confidence or contradictory signals).
	3.	Produce a single actionable trade object for Engine B.
2. Basic MVP Scoring Framework

Inputs
	•	edges: list of active edges from Engine D
	•	Each edge returns:
{"name": "RSI_Bounce", "signal": +1, "weight": 0.5, "confidence": 0.7}
for MVP, confidence = 1 if triggered)

Step 1 – Raw Signal Matrix

For each ticker × edge, collect signals into a table:
Ticker | SMA_Crossover | RSI_Bounce | Bollinger | ATR_Squeeze | Volume_Spike
AAPL   | +1             | 0          | +1         | 0           | +1
MSFT   | 0              | -1         | 0          | 0           | 0
Step 2 – Weighted Score

Compute a weighted sum of all edge opinions:
score = Σ (edge.signal × edge.weight)
Normalize to −1 … +1 by dividing by total weights.

Step 3 – Thresholding

Apply minimum threshold to avoid weak noise:
if score > +0.3 → long
if score < −0.3 → short
else → no trade
. Confidence Calibration (Optional Upgrade)

Later, you can scale each edge’s signal by its historical accuracy or recent Sharpe (from Engine D):
adjusted_weight = base_weight × edge.performance_factor
That’s how you move toward adaptive, self-learning weighting without rewriting Engine A.

⸻

4. Handling Conflicts

Same Ticker, Opposite Edges

If some edges say long and others say short:
	•	Compute net score (weighted average).
	•	If absolute value < threshold → cancel trade (“uncertain zone”).
	•	Optionally, keep a conflict flag for Cockpit diagnostics.

Different Timeframes

If an hourly and daily edge disagree:
	•	Weight higher-timeframe edges more heavily (e.g., ×1.5 multiplier).
	•	MVP can skip this until you add multi-timeframe support.

⸻

5. Final Trade Object

After scoring, Engine A outputs one unified trade per ticker:
{
  "ticker": "AAPL",
  "side": "long",
  "confidence_score": 0.72,
  "entry_price": 175.20,
  "edges_triggered": ["SMA_Crossover", "Bollinger_Breakout", "Volume_Spike"],
  "score_components": {
      "SMA_Crossover": 0.6,
      "Bollinger_Breakout": 0.55,
      "Volume_Spike": 0.45
  }
}
That structure keeps things transparent for Engine B and the Cockpit.

Excellent choice — the confidence formula is literally the mathematical heart of Engine A.
It determines how your system turns multiple edges, signals, and contextual factors into a single probabilistic confidence score for each trade.

Let’s design it step-by-step so it’s simple enough to use in the MVP but flexible enough to evolve into a sophisticated hybrid weighting system later.

⸻

🧠 1. Purpose of the Confidence Formula

Convert raw edge signals (+1, −1, 0) and their characteristics (weights, accuracy, volatility, etc.)
into one unified number between −1 → +1 (direction + strength) or optionally 0 → 1 (probability).

⸻

⚙️ 2. Core Ingredients
Symbol	Meaning	Source
s_i	Edge signal (+1, −1, 0)	Engine A edge output
w_i	Base weight (importance of edge)	Config / Engine D
a_i	Recent accuracy or Sharpe scaling	Engine D feedback
c_i	Raw confidence (if edge outputs magnitude)	Edge itself
r_i	Market-regime multiplier (1 ± δ)	Regime detector
v_i	Volatility or liquidity penalty	DataManager stats
🧩 3. MVP-Level Formula

For now, keep it simple:

\text{score} = \frac{\sum_i s_i \times w_i}{\sum_i |w_i|}
	•	Output range: −1 → +1
	•	Threshold:  |\text{score}| ≥ 0.3 → trade
	•	Side: sign(score)

⸻

🚀 4. Enhanced (Adaptive) Version

Once Engine D supplies performance feedback:

\text{score} =
\frac{\sum_i \left(s_i \times w_i \times a_i\right)}
{\sum_i |w_i \times a_i|}

where
a_i = 1 + k \times (\text{edge Sharpe}_i / \text{avg Sharpe})
k = sensitivity (e.g., 0.2–0.5)

Effect: better-performing edges get louder voices automatically.

⸻

🌦️ 5. Context-Aware (Hybrid) Formula

To include volatility, regime, and liquidity awareness:

\text{score} =
\frac{
\sum_i
\left(
s_i
\times
w_i
\times
a_i
\times
r_i
\times
c_i
\times
v_i
\right)
}{
\sum_i |w_i \times a_i|}

Where typical factors:
	•	r_i < 1 in high-volatility “risk-off” regimes.
	•	v_i < 1 if liquidity is thin (reduce conviction).
	•	c_i defaults = 1 if edge doesn’t output its own confidence.

⸻

📈 6. Normalization & Conversion to Probability

If Engine B prefers a probability-style confidence (0–1):

P_{\text{long}} = \frac{1 + \text{score}}{2}

Then:
	•	P_{\text{short}} = 1 - P_{\text{long}}
	•	Or map with a sigmoid for smoother scaling:
P = 1/(1+e^{-k\text{score}})

⸻

💬 7. Worked Example
Edge	sᵢ	wᵢ	aᵢ	rᵢ	vᵢ
SMA Crossover	+1	0.6	1.1	1.0	1.0
RSI Bounce	0	0.5	1.0	1.0	1.0
Volume Spike	+1	0.4	0.8	1.1	0.9
\text{score} =
\frac{(+1×0.6×1.1) + (0×0.5×1.0) + (+1×0.4×0.8×1.1×0.9)}
{0.6×1.1 + 0.5×1.0 + 0.4×0.8×1.1×0.9}
≈ +0.72

→ Strong long with confidence ≈ 72 %.

Stage	Formula	Purpose
MVP	Σ(s × w)/Σ	w
Adaptive	Σ(s × w × a)/Σ	w × a
Hybrid Contextual	Σ(s × w × a × r × v × c)/Σ	w × a

et’s design a regime-aware layer for Engine A that detects market regimes (trend vs mean-revert, high vs low vol, risk-on vs risk-off, liquidity regimes) and adapts edge weights / activation accordingly. I’ll give you: the set of regimes to detect, concrete indicators/metrics, mapping rules to change weights (and the confidence formula), pseudocode, tuning/testing guidance, monitoring, and pitfalls to avoid.

⸻

1) Which regimes to detect (practical shortlist)

Start with a few that matter most for swing trading:
	1.	Trend vs Mean-Reversion (TR/ MR) — decides whether momentum/trend edges should be upweighted vs mean-reversion edges.
	2.	Volatility Regime (High / Low) — changes sizing and whether breakouts are meaningful.
	3.	Risk Sentiment (Risk-on / Risk-off) — macro sentiment; affects exposure limits (optional).
	4.	Liquidity Regime (Liquid / Thin) — affects slippage penalty and whether to trade smaller names.

You can expand later (e.g., intra-day vs overnight, sector rotation) but these 4 give most practical value.

⸻

2) Concrete detectors & metrics (easy → robust)

Use rolling statistics computed on the same timeframe as your edges (1H / 1D).

Trend vs Mean-Reversion
	•	ADX (Average Directional Index) on price (e.g., 14-period). ADX > 25 → trending; ADX < 20 → non-trending.
	•	Momentum slope: slope of SMA(20) over last N bars (positive slope = trending).
	•	Combine ADX and momentum for robustness.

Volatility Regime
	•	ATR / rolling ATR percentile (14 or 21): ATR > 75th percentile → high-vol; < 25th → low-vol.
	•	Or realized volatility: std(log returns) over 20 periods.

Risk Sentiment
	•	Market breadth proxy: SPY vs equal-weight ETF correlation, or % stocks > 50-day MA; or VIX if available.
	•	Simpler: SPY return over last 10 days > 0 → risk-on; else risk-off.

Liquidity Regime
	•	Relative Volume: current volume / 20-period average volume for symbol.
	•	% of ADV used threshold for filtering.
	•	Use market-wide liquidity by average bid-ask (if you have it) or by ETF volume spikes.

⸻

3) How to convert a detector into numeric regime factors

We want continuous, not binary, signals so weight changes are smooth.

Example mapping functions (output range ~[0.5, 1.5] or [0.0, 1.5]):
	•	Trend factor R_{trend}:
	•	compute ADX_norm = (ADX - 10) / (40) clipped to [0,1]
	•	R_{trend} = 0.8 + 0.7 \times ADX\_norm  → ranges ~0.8–1.5
	•	Mean-reversion factor R_{mr}:
	•	set R_{mr} = 2.0 - R_{trend} (so if trend strong, mr suppressed)
	•	Volatility factor R_{vol}:
	•	vol_pct = percentile of ATR over past 252 days
	•	R_{vol} = 1.0 + (vol\_pct - 0.5) \times 1.0 → ranges ~0.5–1.5
	•	Liquidity penalty P_{liq}:
	•	if rel_volume < 0.5 → P_{liq} = 0.5, else P_{liq} = 1.0
	•	or continuous: P_{liq} = \min(1.0, \max(0.3, rel\_vol))

These are just examples — Engine D will tune via backtest.

⸻

4) How to apply regime factors to the confidence formula

Recall the (hybrid) confidence score:

\text{score} = \frac{\sum_i s_i \times w_i \times a_i \times r_i \times v_i \times c_i}{\sum_i |w_i \times a_i|}

You will now compute per-edge regime multipliers r_i as a combination of the relevant regime factors. Example rules:
	•	If edge is trend-oriented (SMA crossover, breakout):
r_i = R_{trend} \times P_{liq}
	•	If edge is mean-reversion (RSI bounce):
r_i = R_{mr} \times P_{liq}
	•	For volatility-sensitive edges (Bollinger breakout, ATR squeeze):
r_i = f_{\text{edge}}(R_{vol}) \times P_{liq}
where f_{\text{edge}} might upweight breakouts in high vol or upweight squeezes in low vol depending on your hypothesis.

So per edge you pick the formula for mapping regimes → r_i.

⸻

5) Putting it together — an end-to-end recipe
	1.	Compute regime metrics once per update:
	•	ADX, SMA slope, ATR percentile, SPY 10-day return, rel_volume for each ticker.
	2.	Map to continuous factors:

C — Execution model, slippage, and costs (realistic simulation)

A credible backtest must simulate execution realism:

1) Order types simulated
	•	Market order: filled at next available price (use next candle open or VWAP approximation).
	•	Limit order: filled if the intraperiod high/low reaches the limit.
	•	Stop order: converted to market on trigger (simulate slippage).

2) Slippage model (practical retail-level)
	•	Two components: fixed per-trade cost (commissions) + impact/slippage as function of order size vs liquidity.
	•	Example simple model:
	•	slippage_pct = base_slippage + impact_coeff * (order_notional / avg_daily_volume)
	•	base_slippage: e.g., 0.01% (1 bps) for liquid ETFs/stocks
	•	impact_coeff: e.g., 0.5  (so if you trade 1% of ADV, slippage = 0.005% + base)
	•	For very small accounts, ADV fraction is tiny so slippage small. For larger simulated accounts increase impact.

3) Commission / fees
	•	Per-share or per-trade fee: include realistic broker fee (e.g., $0.0003 per share or $0 for commission-free brokers).
	•	For options/futures include contract fees and exchange fees (later).

4) Fill price simulation (practical)
	•	For market buys: use fill_price = next_candle_open * (1 + slippage_pct)
	•	For market sells: fill_price = next_candle_open * (1 - slippage_pct)
	•	For limit orders: if limit between open & high/low, use limit price; else no fill.

5) Partial fills & order splitting (advanced)
	•	If order_size > fraction threshold of ADV (e.g., >1% ADV), split into N child orders across subsequent candles and accumulate slippage per child.
	•	For MVP you can approximate partial fill by scaling fill probability.

6) Transaction-level logging
	•	Log: order_id, fill_price, qty, slippage_pct, commission, timestamp, edges_triggered, score, regime_state.

1. What an edge is

An edge is anything in trading that gives you a statistical advantage over the market, such that, over many trades, it produces positive expected value (EV).

Formally:

\text{Expected Return per Trade} = (\text{Win Rate} \times \text{Avg Win}) - (\text{Loss Rate} \times \text{Avg Loss}) > 0

If this formula is positive over a large sample, you have an edge.

In plain English:
	•	It’s a repeatable factor that lets you consistently make money while controlling risk.
	•	Without an edge, trading is essentially gambling.

⸻

2. Types of edges

A. Price / technical edges
	•	Patterns in price that are statistically more likely to work than random chance.
	•	Breakouts, mean reversion, trend continuation
	•	Technical indicators: moving averages, RSI, Bollinger Bands, VWAP
	•	Example: Buying a stock when its price crosses above the 50-day MA might have a historical 60% chance of rising over the next week.

B. Statistical / quantitative edges
	•	Based on historical data and probability:
	•	Seasonal effects, earnings surprises, option flow patterns
	•	Correlations, factor models, volatility anomalies
	•	Example: Certain small-cap stocks tend to revert to mean after a 3-day drop.

C. Event-driven / informational edges
	•	Using information not yet fully priced in by the market:
	•	News, tweets, sentiment, insider buying, macroeconomic data
	•	Example: Stock jumps after unexpectedly positive earnings announcements.

D. Behavioral / psychological edges
	•	Markets are made by humans — some patterns repeat due to psychology:
	•	Overreaction to news → mean reversion
	•	Herding behavior → momentum moves

E. Execution edges
	•	Edges in how you place trades, not just what you trade:
	•	Reducing slippage, timing entries better, using liquidity smartly
	•	Even a 0.1–0.2% edge per trade from execution improvements can compound significantly over thousands of trades

3. Why edges are critical
	1.	Without an edge, you are gambling. You might win occasionally, but the law of large numbers ensures losses dominate over time.
	2.	Edges allow compounding. A mechanical system only works if it has a repeatable advantage.
	3.	Different edges scale differently. Small-cap edges might not scale to $100k+ without affecting prices; liquid ETFs scale better but edges are smaller.

4. Examples of simple edges for swing trading
	•	Buy dips when RSI < 30 and trend is positive → historically 60–65% success rate
	•	Buy after 3 consecutive down days if stock has high liquidity → mean reversion
	•	Trade ETFs after overnight gap fills → historically profitable ~55–60% of the time
	•	Momentum breakout of high-volume stocks → average move exceeds stop-loss by 2:1

The key: each edge is quantifiable, repeatable, and has positive expected value over time, not just intuition.

Important Notes About Edges
	1.	Edges must be tested statistically — don’t rely on intuition. Backtest over thousands of trades.
	2.	Edges degrade over time — if many traders exploit the same anomaly, returns diminish.
	3.	Combine edges for better expectancy — a single edge with 52–55% win rate may not be enough; combining complementary edges can improve the overall edge.
	4.	Edge size vs frequency — some edges are small but frequent (price patterns), others are big but rare (earnings surprises).

from pathlib import Path
import pandas as pd

from datetime import datetime

class EdgeResearchDB:
    def __init__(self, db_path="data/research/edge_results.parquet"):
        self.db_path = Path(db_path)
        self.db_path.parent.mkdir(parents=True, exist_ok=True)
        self.df = pd.read_parquet(self.db_path) if self.db_path.exists() else pd.DataFrame()

    def append_run(self, results_csv: str):
        new = pd.read_csv(results_csv)
        new["timestamp"] = datetime.utcnow().isoformat()
        self.df = pd.concat([self.df, new], ignore_index=True)
        self.save()

    def save(self):
        self.df.to_parquet(self.db_path, index=False)

    def rank_edges(self):
        if self.df.empty:
            return pd.DataFrame()
        grouped = (
            self.df.groupby("edge")[["sharpe", "cagr_pct", "max_drawdown_pct"]]
            .mean()
            .reset_index()
        )
        grouped["score"] = (
            0.5 * grouped["sharpe"].fillna(0)
            + 0.3 * (grouped["cagr_pct"].fillna(0) / 10)
            - 0.2 * grouped["max_drawdown_pct"].abs()
        )
        grouped = grouped.sort_values("score", ascending=False)
        return grouped"
